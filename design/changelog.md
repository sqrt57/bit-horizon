# Changelog

Resolved decisions and notable completed work. See [backlog.md](backlog.md) for
open questions and pending tasks.

- **Implementation platform** — switched from Android-native (Jetpack Compose) to **Compose Multiplatform**, targeting Android and Desktop. Game logic is pure math with no platform-specific UI needs, so the switch costs little beyond Kotlin Multiplatform project setup (`commonMain`/`androidMain`/`desktopMain` source sets).
- **iOS support** — deferred indefinitely: Kotlin/Native's iOS target requires Xcode on macOS to build, which isn't available in this environment (Windows). The KMP module structure keeps the door open to adding an `iosMain` source set later if Mac access becomes available.
- **Persistence approach** — using shared Jetpack `DataStore` in `commonMain` for now: save state is a handful of scalars (Data Bits, three tier counts, Tachyon Particles, a timestamp), which fits DataStore's key-value model. Revisit with **SQLDelight** (multiplatform SQLite, native drivers for Android/iOS/Desktop) if the design grows something relational to query — e.g. persisted Log Hub history, multiple save slots, or an achievements table.
- **Large-number representation** — `Double` alone loses precision well before late-game idle-game scales. Decision: a hand-rolled Kotlin port of break_infinity.js, not a general `BigDecimal` library — rejected `BreakInfinity.java` (JVM-only, not usable from the iOS target) and `kotlin-multiplatform-bignum` (arbitrary-precision, KMP-safe, but heavier than idle games need).
- **break_infinity.js port** — `Decimal` (`composeApp/src/commonMain/kotlin/com/cosmic/bithorizon/math/`) ported with full API parity (arithmetic, comparisons incl. tolerance variants, rounding, log/pow/sqrt/cbrt, string formatting, factorial/hyperbolic/joke functions) plus the `math.ts` series-purchase helpers (`affordGeometricSeries` etc.). Immutable design (not upstream's mutate-during-construction pattern); exponent kept as `Double` to match JS `Number` semantics exactly (`EXP_LIMIT = 9e15` exceeds `Int` range). String formatting (`toString`/`toFixed`/`toExponential`/`toPrecision`) is hand-rolled directly from the `(mantissa, exponent)` pair rather than delegating to `Double.toString()`, since JVM and JS switch to scientific notation at different magnitude thresholds and `String.format`/`java.text.*` aren't available from `commonMain` anyway — keeps the door open for a future `iosMain` target. Dropped upstream's `decimal-pool.ts` caching (JS-GC-pressure optimization not worth it on JVM) and the companion-mirrored static methods (e.g. `Decimal.add(a, b)`) since Kotlin call sites just use `a.add(b)`. Test suite ported to `commonTest` (construction round-tripping, comparison sign correctness, string-formatting at exponent boundaries, `toDouble()` overflow/underflow, mantissa rounding) using `kotlin.test`.
- **Project scaffold** — Kotlin Multiplatform + Compose Multiplatform Gradle project created: `composeApp` module with `commonMain`/`androidMain`/`desktopMain` source sets. Versions: Gradle 8.14, AGP 8.13.0, Kotlin 2.2.20, Compose Multiplatform 1.11.1, compileSdk/targetSdk 36, minSdk 26 (per GDD), JVM target 17. Both `assembleDebug` (Android) and `compileKotlinDesktop` (Desktop) verified building successfully.
- **Game state model** — `GameState` (`composeApp/src/commonMain/kotlin/com/cosmic/bithorizon/game/GameState.kt`) added: immutable data class holding Data Bits, the three tier counts, and Tachyon Particles as `Decimal`. Buy-cost math (`costForNext`, `costFor`, `maxAffordable`, `buy`, `buyMax`) is built directly on `DecimalMath`'s `affordGeometricSeries`/`sumGeometricSeries` rather than reimplementing the series math. Cascading payment currency (Tier 2 costs Tier 1's count, etc.) is modeled per-tier via `paymentBalance(tier)` rather than a generic resource graph, since there are only three fixed tiers and the GDD doesn't anticipate more.
- **Delta-ticker production loop** — `GameState.tick(deltaSeconds)` added, implementing the GDD's `ΔTier_(n-1) = Tier_n × Multiplier_n × Δt` formula at the spec'd `Δt = 0.1s` (`TICK_DELTA_SECONDS`). `TierConfig` gained a `baseOutputPerSecond` field (all tiers currently `1.0`, but per-tier for when the GDD's Base Output column diverges). Each Tachyon Particle applies its permanent +100% efficiency bonus to every production channel, not just the final Data Bits stream, per the prestige section's "all production output channels" wording. All three tiers' output is read from the state *before* any tier is updated, so a single tick can't cascade production through multiple tiers at once — order-independent by construction rather than relying on iteration order. Deliberately left as a pure function with no real-time driver (no coroutine loop calling it on a timer): there's no state holder or UI to host that loop yet, so it's expected to land as a `LaunchedEffect` when `App.kt`'s zones are wired up.
- **Horizon Shift prestige logic** — `GameState.canHorizonShift()`/`horizonShift()` added per the GDD's prestige section. `HORIZON_SHIFT_THRESHOLD` is `10,000` Data Bits (`Decimal.fromInt`, not `fromDouble(1e4)`, to avoid a needless float round-trip for an exact integer threshold). `horizonShift()` wipes Data Bits and all three tier counts and increments (not resets) Tachyon Particles by 1, so particles accumulate across repeated shifts — the permanent +100%/particle efficiency bonus was already wired into `tick()` when it was added, so no changes were needed there. Below the threshold, `horizonShift()` returns the state unchanged rather than throwing, consistent with `buy()`'s no-op-on-unaffordable pattern. No UI trigger yet (the GDD's `[HORIZON SHIFT]` button morph is part of the Executive Core zone, still a TODO placeholder in `App.kt`).
- **App.kt wired to GameState** — the four GDD-specified zones replaced the `App.kt` TODO placeholders: Zone 1 shows Data Bits (`toExponential(2)`) and Bits/sec (new `GameState.dataBitsPerSecond()` helper, the GDD's Prestige Modifier formula exposed as a queryable value instead of only being computed inline inside `tick()`); Zone 2 is a `LazyColumn` log of purchase/prestige events, newest first, capped at 100 entries; Zone 3 lists the three tiers with owned count, next cost, and a "Buy Max" button (disabled via `maxAffordable(tier) <= 0`, calling `GameState.buyMax`); Zone 4 is a single button that reads "MANUAL SCAN" and calls the new `GameState.manualScan()` below the Horizon Shift threshold, morphing to "[ HORIZON SHIFT ]" and calling `horizonShift()` at/above it, per the GDD's "execution button morphs" spec. A `LaunchedEffect(Unit)` drives the game loop with `delay(TICK_DELTA_SECONDS * 1000) ; state = state.tick()` — the real-time driver `tick()` was deliberately left without when it was added. `manualScan()`'s yield is a placeholder (see `design/backlog.md`'s "Manual Scan yield" open question) since the GDD doesn't specify one. Verified by launching the desktop app (`:composeApp:run`) and driving it interactively: Buy Max purchases matched the geometric-series cost formula exactly across multiple sequential buys, Bits/sec tracked owned Telemetry Sensors correctly, and lower-tier buttons correctly disabled when unaffordable.
